  SystemError : Class		 
  shortTermScheduler : Class

Reads jobs from the ready queue filled by the long-term scheduler

and sorts the queue by the jobs' priority. The short-term scheduler

then waits until the CPU requests a job. When the CPU sends its request

the short-term scheduler takes the next job in the ready queue and passes

it to the CPU along with its metadata to be processed

 
  RamMemory : Class

Implementaion of the OS's memory.

Contains methods to add/modify data in

memory (via the memory manager). This class

is not directly accessed.

 
  ProcessControlBlock : Class

Contains all information related to each

process

 
  PCB_block : Class

This is a template for an individual PCB entry and

its associated attributes.

 
  pageTableEntry : Class

Template for a page table entry

  PageFault : Class

Defines a page fault exception

 
  OSToolkit : Class

This class is just a set of tools used by

other components in the OS.

 
  OSDriver : Class

Core or main thread of the OS Simulator. Calls the

loader to load the user programs and then calls the short-term scheduler

to assign a job to the CPU from the ready queue created by the

long-term scheduler.

 
  MemoryManager : Class

All reading and writing to and from disk and ram

is done so via the memory manager

 
  LongTermScheduler : Class

Reads the job entries from the PCB and places the

associated job data into RAM, updates the job's

associated PCB entry and places the job into the

ready queue

 
  Loader : Class

Reads the data file and adds the hex

entries to the disk.

The job metadata read creates a PCB object

and once the associated data metadata is read,

the PCB is updated and placed in the PCB array.

 
  FrameTableEntry : Class

Template for a frame table entry and

its associated attributes

 
  DiskMemory : Class

Implementation of the OS's disk.

Contains methods to add/modify disk

data via the memory manager, this class

is not directly accessed.

 
  CPU : Class

Reads the job and associated data from RAM via the

Memory manager. The data is converted to a 32-bit string,

decoded and executed. Upon completion of execution, or in the event

of an error (causing the CPU to halt), control is passed back

to the driver.

Generates cpulog.txt in project directory for testing
